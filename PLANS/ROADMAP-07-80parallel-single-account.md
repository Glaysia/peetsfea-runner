# ROADMAP 07: 1계정 10잡 x 8window = 80 병렬

## 목표/범위

목표:

1. 단일 계정에서 Slurm 잡 10개를 동시 운영
2. 각 잡에서 screen 1세션 + window 8개 실행
3. 총 80 병렬 처리 안정화

## 아키텍처 변경점

1. 잡 슬롯 관리자(`job slot manager`) 도입
2. 잡별 독립 session naming 규칙 도입
3. 계정 단위 큐/백프레셔 제어 계층 추가

## 인터페이스/API/스키마 변경

1. 내부 스케줄링 파라미터:
   - `max_jobs_per_account=10`
   - `windows_per_job=8`
2. 잡 메타데이터 테이블(또는 파일) 추가:
   - `job_id`, `session_name`, `state`, `started_at`, `finished_at`

## 실행 플로우

1. 큐에서 작업 배치 추출
2. 슬롯 여유에 맞춰 잡 제출
3. 각 잡 내 8 window 실행
4. 잡 완료 시 결과 회수/정리
5. 슬롯 해제 후 다음 잡 할당

## 리소스 정책(코어/라이선스/계정/잡)

1. 코어: 잡당 32, 계정당 최대 320
2. 라이선스: 계정당 최대 80 동시 사용(정책 상한 적용)
3. 계정: 1
4. 잡: 최대 10

## 실패 처리 및 복구 절차

1. 잡 단위 격리: 특정 잡 실패가 나머지 잡 진행을 막지 않음
2. 잡 재시도 상한 1회, 실패 누적 시 격리 큐 이동
3. screen/session orphan 감지 시 강제 정리

## 테스트/수용 기준

1. 80 병렬에서 처리량/실패율 지표 수집
2. 백프레셔 동작 검증(슬롯 초과 제출 금지)
3. 장애 잡 격리 및 재시도 동작 검증
4. 원격 정리 누락 0건

## 운영 체크리스트

1. 잡 슬롯 상한 설정 확인
2. 잡/세션 이름 충돌 없음 확인
3. 실패 잡 격리 큐 점검

## 롤백 기준

1. 슬롯 관리자 불안정 시 06 단계(1잡 8window)로 롤백
